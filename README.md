# Micro-Ordering

A production-ready **microservices ordering system** built with NestJS, demonstrating advanced backend patterns including the Transactional Outbox Pattern, Idempotency handling, and JWT-based distributed authentication.

---

## Overview

Micro-Ordering simulates a real-world e-commerce backend where placing an order triggers a reliable billing pipeline — even in the face of network failures, duplicate requests, or broker restarts. The system is designed around **reliability** and **consistency** as first-class concerns.

## Services

### Orders Service (`apps/orders`)
Handles order creation via REST API. Exposes:
- `POST /orders` — Create a new order (requires JWT + Idempotency-Key header)
- `GET /orders` — List all orders

### Auth Service (`apps/auth`)
Handles user registration and authentication. Issues JWT tokens stored as `httpOnly` cookies. Validates tokens for other services via RabbitMQ (`validate_user` message pattern).

### Billing Service (`apps/billing`)
Consumes `order_created` events from RabbitMQ and creates invoices in MongoDB. Fully idempotent — safe to receive the same event multiple times.

---

## Key Patterns & Design Decisions

### Transactional Outbox Pattern
Order creation and outbox event insertion happen in a **single MongoDB transaction**. A scheduler (`OutboxPublisherService`) polls pending events every 5 seconds and publishes them to RabbitMQ — guaranteeing at-least-once delivery even if the broker is temporarily unavailable.

```
MongoDB Transaction
├── INSERT orders         (order data)
└── INSERT outbox_events  (status: PENDING)

OutboxPublisher (every 5s)
└── SELECT pending event
    ├── PUBLISH to RabbitMQ → status: PUBLISHED
    └── on failure          → status: PENDING, retryCount++  (max 3 retries)
```

### Idempotency — HTTP Layer (Orders Service)
Every `POST /orders` request requires an `Idempotency-Key` header (UUID generated by the client). The server stores the key alongside the response. Duplicate requests with the same key return the original response without creating a new order.

```
First request  → Idempotency-Key: uuid-123 → 201 Created  (new order)
Retry request  → Idempotency-Key: uuid-123 → 200 OK       (same order, no duplicate)
Missing header →                            → 400 Bad Request
```

### Idempotency — Event Layer (Billing Service)
The `Invoice` schema enforces a **unique index on `orderId`**. If RabbitMQ redelivers an `order_created` event, the duplicate insert fails with MongoDB E11000 — which is caught and silently ignored. No separate deduplication table needed.

### Distributed JWT Authentication
`JwtAuthGuard` sends the cookie token to the Auth Service via RabbitMQ for validation. This means **no shared JWT secret** between services — auth logic stays centralized.

## Data Flow — Place an Order

```
1. Client sends POST /orders
   Headers: { Cookie: Authentication=<jwt>, Idempotency-Key: <uuid> }

2. JwtAuthGuard → sends token to Auth Service via RabbitMQ
   Auth Service validates → returns user payload

3. IdempotencyGuard → checks idempotency_records collection
   - Key exists? → return cached response (200 OK)
   - Key missing? → continue

4. OrdersService.createOrder() — single MongoDB transaction:
   ├── INSERT orders
   ├── INSERT outbox_events (status: PENDING)
   └── INSERT idempotency_records (key + response)

5. OutboxPublisherService (every 5s):
   └── Picks PENDING event → emits 'order_created' to RabbitMQ

6. BillingService.bill():
   ├── Check if invoice already exists for orderId
   ├── No  → INSERT invoices (status: ISSUED)
   └── Yes → log warning, skip (idempotent)
```

## Getting Started

### Prerequisites
- Node.js 20+
- MongoDB 6 (running locally with replica set `rs0`)
- RabbitMQ

**Optional:** Docker & Docker Compose (for containerized setup)

### Installation

```bash
git clone https://github.com/mtdamir/Micro-Ordering.git
cd Micro-Ordering
npm install
```

### Environment Variables

**`apps/orders/.env`**
```env
MONGO_URI=mongodb://localhost:27017/orders?replicaSet=rs0&directConnection=true
PORT=3000
RMQ_URL=amqp://guest:guest@localhost:5672
RMQ_QUEUE_BILLING_QUEUE=billing_queue
RABBIT_MQ_AUTH_QUEUE=auth_queue
```

**`apps/auth/.env`**
```env
MONGO_URI=mongodb://localhost:27017/auth?replicaSet=rs0&directConnection=true
JWT_SECRET=your_secret_here
JWT_EXPIRATION=3600
RMQ_URL=amqp://guest:guest@localhost:5672
RABBIT_MQ_AUTH_QUEUE=auth_queue
```

**`apps/billing/.env`**
```env
MONGO_URI=mongodb://localhost:27017/billing?replicaSet=rs0&directConnection=true
RMQ_URL=amqp://guest:guest@localhost:5672
RMQ_QUEUE_BILLING_QUEUE=billing_queue
RABBIT_MQ_AUTH_QUEUE=auth_queue
```

> **Note:** `directConnection=true` is required when using a single-node replica set locally — it prevents Mongoose from trying to discover other replica set members.

### MongoDB — Verify Replica Set

Before starting the services, make sure MongoDB is running with replica set enabled:

```bash
mongosh --eval "rs.status()"
# Expected: set: 'rs0', stateStr: 'PRIMARY'
```

If not initialized yet:

```bash
mongosh --eval "rs.initiate({ _id: 'rs0', members: [{ _id: 0, host: 'localhost:27017' }] })"
```

### Run with Docker

```bash
docker-compose up --build
```

> Docker Compose uses `mongo:7.0` official image with a single-node replica set (`rs0`) initialized via `mongo-init` service. RabbitMQ management UI is available at `http://localhost:15672` (guest/guest).

### Run in Development

```bash
# Terminal 1
npm run start:dev auth

# Terminal 2
npm run start:dev orders

# Terminal 3
npm run start:dev billing
```
